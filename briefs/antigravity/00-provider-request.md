## Preferred Direction

Use this section to provide a possible direction, but the model should research and improve it if needed.

Main trap candidate:
Agent autonomy inside a real codebase creates review and control risk. The agent can modify files, add dependencies, change architecture, generate tests, move logic, and attempt fixes faster than a beginner can understand the consequences.

Secondary trap candidate:
Visible agent activity can feel like progress even when the project is drifting. A beginner may mistake completed tasks, generated plans, passing-looking artifacts, or long diffs for real forward motion without checking whether the app is simpler, safer, cheaper, or easier to maintain.

The final brief should make clear:

- Google Antigravity is not the problem.
- The danger is handing an agent more authority than your review process can handle.
- Agent speed is valuable, but only if the builder controls task size, checks diffs, understands changed files, tests behavior, and knows how to roll back.
- A beginner should not measure progress by how much the agent changed. They should measure progress by whether the app became more understandable, testable, and maintainable.
- The pamphlet should help first-time solo builders use agentic development without letting the agent outrun their judgment.
