## Provider Context

- Vercel is a platform for building, deploying, scaling, and managing web applications, including AI-powered applications and agentic workloads.
- Vercel can make deployment feel extremely simple: connect a repo, push code, get a live URL, preview changes, and promote to production.
- That simplicity is valuable, but it creates a beginner trap: first-time solo builders may treat a successful deployment as proof that the app is finished, safe, cheap, and ready for users.
- Vercel introduces real production responsibilities: environment variables, preview vs production environments, runtime logs, usage limits, usage-based resources, build behavior, function execution, analytics, image optimization, configuration, domains, and monitoring.
- Avoid exact prices, current quotas, credit amounts, plan names beyond broad categories, or detailed limits unless verified at publication time.
- Be careful to distinguish Vercel platform costs/responsibilities from third-party API costs. Do not imply Vercel is responsible for every cost inside an AI app.
- Do not imply a Hobby/free user automatically starts paying when limits are hit. Keep wording conservative: limits, caps, overages, usage-based resources, and paid-plan behavior vary by plan and current terms.

## Preferred Direction

Use this section as the likely direction, but the model may improve it if needed.

Main trap candidate:
Deployment feels like the finish line before the beginner understands production responsibility. A live URL can make a project feel complete, but it actually means the app is now reachable, observable, configurable, and responsible for real usage.

Secondary trap candidate:
Preview and production environments blur together. A beginner may test in a preview branch, then push to production without understanding environment variables, secrets, logs, configuration, domains, usage behavior, or what changes when real users arrive.

The final brief should make clear:

- Vercel is not the problem.
- The danger is treating deploy as done.
- A live URL is not proof that the app is finished. It is proof that the app is reachable.
- Beginners should understand environment variables, preview vs production, logs, usage categories, domains, and rollback before inviting users.
- The pamphlet should separate Vercel platform responsibility from third-party AI/API/service responsibility.
- The brief should not be anti-deployment. It should teach first-time solo builders to deploy with supervision and guardrails.
