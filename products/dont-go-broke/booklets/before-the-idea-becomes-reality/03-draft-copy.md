# Before the Idea Becomes Reality
## A practical field guide for deciding whether an AI app idea is ready to become a real project

### Series Note

Don’t Go Broke helps builders avoid preventable mistakes in AI-assisted projects.

The goal is not to make building feel harder.

The goal is to help you avoid the expensive parts that happen when a project starts before the decisions are clear.

This booklet focuses on the earliest mistake:

**turning an unclear idea into a real project too soon.**

Later booklets deal with designing the project, reviewing the first AI-built draft, and managing what happens after launch.

This one comes first.

Before the platform.

Before the blueprint.

Before the agent.

Before the build request.

Before the idea becomes reality.

---

### How to Use This Booklet

Read this before you open an AI builder and ask it to make your app.

You do not need a perfect plan.

You do not need startup language.

You do not need a pitch deck, market research report, technical diagram, or product strategy document.

You only need to slow down long enough to answer a few plain questions:

- What is the idea?
- What problem does it solve?
- Who is version one for?
- What counts as version one?
- What is not included yet?
- What would make this project too vague, too expensive, or too large?
- Is this ready to become a real project?

This booklet is not here to kill your idea.

It is here to protect it from becoming expensive before it becomes clear.

---

## 1. Introduction: Why This Booklet Exists

AI has made it easier than ever to start building.

That is useful.

It is also dangerous.

A few years ago, an unclear app idea usually stayed unclear for a while. You might talk about it, sketch it, write notes, or imagine what it could become. There was friction between the idea and the build.

Now that friction is much lower.

You can open an AI builder, describe your idea, and watch it generate screens, files, features, database assumptions, flows, and copy before you have decided what the project actually is.

That can feel amazing.

It can also make the idea feel more ready than it really is.

The danger is not that you are excited.

You should be excited.

The danger is that excitement can turn into a build request before the idea has earned the right to become a project.

Once AI starts building, the idea begins to create gravity.

Now there are files.

Now there are features.

Now there is something to fix.

Now there are things to explain.

Now there are decisions you did not realize you had made.

Now there is a demo you may feel attached to.

Now stopping feels like waste, even if the project was never clearly defined.

That is how a vague idea becomes a real project before you are ready to manage it.

This booklet creates a pause before that happens.

Not a long pause.

Not a formal planning process.

Not a business-school exercise.

Just enough of a pause to decide what must be true before your idea becomes real.

Before you ask AI to build anything, answer this:

```text
What am I about to ask AI to build?
```

That question is only the beginning.

Because the honest answer may be:

```text
I know what I am excited about.
But I do not yet know what the project is.
```

That is not failure.

That is the exact moment this booklet is for.

## 2. Why AI Makes Ideas Feel Ready Too Soon

AI can make an idea look real very quickly.

That is part of the appeal.

You type a description. The tool responds. Something appears. A page. A button. A dashboard. A workflow. A demo. A first version of something that did not exist a few minutes ago.

It feels like progress.

Sometimes it is progress.

But visible output is not the same as a good project decision.

A demo is not proof that the idea deserves to become a real product.

A working screen is not proof that the problem matters.

A generated feature is not proof that the feature belongs.

A fast build is not proof that the project is clear.

AI is very good at making vague ideas feel more concrete. That can be helpful when you are exploring. But it can also trick you into believing the idea is ready because something visible exists.

That visible thing creates emotional pressure.

Once you see it, you may start reacting to it:

- “Can you make the dashboard better?”
- “Add login.”
- “Add profiles.”
- “Add payments.”
- “Make it mobile.”
- “Add admin controls.”
- “Add AI suggestions.”
- “Add a settings page.”
- “Add sharing.”
- “Add notifications.”

Now the project is growing.

But growing around what?

If the original problem was unclear, every new feature makes the project harder to understand.

If the first user was unclear, every new screen tries to serve too many people.

If version one was never defined, every addition feels reasonable.

AI can keep going.

That does not mean you should.

A quick demo is like a sketch. It can help you see possibilities. But it is not a permit to pour the foundation.

Before you trust the excitement of visible progress, ask:

```text
What would make this idea worth building if AI had not already generated 
anything?
```

That question matters because it separates the value of the idea from the thrill of seeing something appear.

Maybe the idea is worth building.

Maybe it needs more thought.

Maybe it should stay small.

Maybe it should wait.

The answer should come from the problem, the user, and the first version.

Not from the fact that AI made something quickly.

## 3. The Difference Between an Idea and a Project

An idea says:

```text
This could be something.
```

A project says:

```text
This is what we are building first.
```

Those are not the same.

An idea can be loose. It can be exciting, messy, unfinished, and full of possibilities.

A project is different.

A project creates obligations.

A project has scope.

A project has files.

A project has decisions.

A project has costs.

A project has things to review.

A project has things that can break, drift, expand, and confuse you.

That does not mean projects are bad. It means projects are real.

Before your idea becomes a project, you need a boundary.

The boundary does not need to be fancy. It only needs to be clear enough to protect you from building everything at once.

A project should have answers to four basic questions:

```text
What problem does this solve?
Who is version one for?
What does version one include?
What is not included yet?
```

If you cannot answer those questions yet, you may still have a good idea.

But you may not have a project yet.

That distinction matters because AI does not always know the difference.

If you give AI a vague idea, it may confidently build a vague project.

It may create structure around assumptions you never approved.

It may add features because they seem normal for that type of app.

It may fill in gaps with patterns that look reasonable but do not match what you actually need.

That is not the tool being malicious.

That is what happens when a builder asks for construction before defining the job.

Once the idea becomes a project, it starts pulling in more decisions. That is the gravity.

The question is:

```text
Is this still an idea, or have I already started treating it like a project?
```

A second question may be even more useful:

```text
What have I already committed to before defining the project clearly?
```

Maybe you have already committed to a feature.

Maybe you have already committed to a platform.

Maybe you have already committed to a design direction.

Maybe you have already committed to solving three problems instead of one.

Maybe you have already committed emotionally because a demo exists.

That is okay.

You can still step back.

The goal is not to punish yourself for starting.

The goal is to notice the difference between imagining and committing.

Because once you know where the boundary is, you can decide whether to cross it.

## 4. What Problem Does This Actually Solve?

Many app ideas begin as features.

“I want an app with profiles, reminders, dashboards, AI suggestions, groups, ratings, notifications, and a clean interface.”

That may describe what the app contains.

It does not yet explain what problem it solves.

Features are not the problem.

Features are possible ways to solve the problem.

If the problem is unclear, AI can still build features. That is part of the risk. You can end up with a lot of visible work wrapped around a vague purpose.

Before your idea becomes a project, write the problem in plain language.

Do not try to sound like a founder.

Do not try to sound technical.

Do not try to make it impressive.

Try to make it clear.

Use this sentence:

```text
This idea helps __________ do, avoid, decide, understand, or manage __________ 
because __________.
```

Examples:

```text
This idea helps roommates decide what to eat because group food decisions take 
too long and become annoying.
```

```text
This idea helps a small business owner remember follow-ups because important 
customer tasks get lost during busy weeks.
```

```text
This idea helps students organize project deadlines because assignments, notes, 
and reminders are scattered across too many places.
```

The sentence does not need to be perfect.

It needs to be clear enough to guide decisions.

A weaker version sounds like this:

```text
The app has reminders, profiles, an AI assistant, a dashboard, and sharing.
```

That may be useful later.

But right now it avoids the real question.

The real question is:

```text
What problem would still exist if this app did not?
```

If you can only describe features, the idea is probably not ready to become a project.

That does not mean the idea is bad.

It means the idea still needs a clearer job.

A useful problem statement gives you a way to judge version one.

If the problem is “groups take too long to decide what to eat,” then version one does not need every restaurant feature you can imagine.

It needs to help the group decide faster.

If the problem is “small business follow-ups get lost,” then version one does not need a full business operating system.

It needs to help the owner capture and act on follow-ups.

If the problem is “students lose track of deadlines,” then version one does not need to become an entire school platform.

It needs to help the student see and manage what is due.

The problem protects the project.

It helps you say yes to what matters.

It also helps you say “not yet” to what does not.

Before moving on, write the simplest version you can:

```text
The problem is: __________.
```

Then ask:

```text
Could someone understand this problem without seeing the app?
```

If yes, you are closer.

If no, keep simplifying.

## 5. Who Is Version One Actually For?

Version one is not for everyone.

That can be hard to accept because your idea may eventually help many kinds of people.

That is fine.

Eventually is not version one.

Version one needs a first user.

Not a perfect persona.

Not a market segment.

Not a full audience strategy.

Just a clear answer to this:

```text
Who needs the first useful version of this?
```

If your answer is “everyone,” version one has no anchor.

When version one is for everyone, every feature starts to sound important.

Different people need different things.

Different use cases pull the project in different directions.

Different expectations create different screens, settings, permissions, and workflows.

That is how a small idea becomes large before it becomes clear.

A first user protects you from that.

Use this sentence:

```text
The first version is for __________, who needs help with __________.
```

Examples:

```text
The first version is for a group of friends who need help choosing what to eat 
without debating for half an hour.
```

```text
The first version is for a solo contractor who needs help tracking customer 
follow-ups after each job.
```

```text
The first version is for a student who needs one place to see what assignments 
are due this week.
```

Notice what these examples do.

They do not describe the whole future.

They describe the first person the app must help.

That first user gives version one a test.

If the app helps that person with that problem, version one may be useful.

If it does not, more features will not save it.

The first user also helps you reduce scope.

You may eventually want families, teams, businesses, schools, and creators to use the product.

But version one may only need to serve one of them.

That is not a lack of ambition.

It is a way to keep the first project understandable.

Ask:

```text
If this first user is the only person version one helps, what does it need to do 
well?
```

That answer matters more than the full feature list.

A first version needs someone specific to help.

Without that person, the project can drift in every direction at once.

## 6. What Counts as Version One?

Version one is not the whole dream.

Version one is the smallest useful version of the project.

That word matters:

**useful.**

Not impressive.

Not complete.

Not future-proof.

Not everything you can imagine.

Useful.

Version one should solve one clear problem for one clear first user.

It should be small enough that you can understand it, review it, and control it.

That is especially important when AI is helping build.

If version one is too large, you may not know whether AI built the right thing.

You may not know where the complexity came from.

You may not understand which feature caused which problem.

You may not be able to tell whether the project is drifting away from the original idea.

A smaller version one gives you a better chance to stay in control.

Use this sentence:

```text
The first useful version only needs to let the user __________.
```

Examples:

```text
The first useful version only needs to let the group submit choices and reveal a 
shared decision.
```

```text
The first useful version only needs to let the contractor record a customer 
follow-up and see what needs attention today.
```

```text
The first useful version only needs to let the student enter assignments and see 
what is due this week.
```

These are not complete products.

That is the point.

A first room is not the whole building.

But it can prove that the structure can stand.

Version one should be narrow enough that you can answer:

```text
If this one thing worked, would version one be useful enough to learn from?
```

If the answer is yes, you may have a version-one boundary.

If the answer is no, either the problem is unclear or the version is too thin.

If the answer requires ten more features, version one may be too large or the idea may need more work.

Every added feature increases the load.

More features mean more decisions.

More decisions mean more opportunities for AI to guess.

More guesses mean more review work.

More review work means more places for cost, confusion, and drift to enter.

This does not mean features are bad.

It means every feature should earn its place.

Before the build starts, version one should be something you can explain in one or two sentences.

If you need a long tour to explain version one, it is probably not version one yet.

## 7. What Is Explicitly Not Included Yet?

A good version one needs a “not yet” list.

This may be the most protective page in the whole booklet.

When you are excited, every future feature can feel important.

You may think:

- “We will need accounts.”
- “We will need sharing.”
- “We will need payments.”
- “We will need notifications.”
- “We will need AI recommendations.”
- “We will need admin tools.”
- “We will need analytics.”
- “We will need settings.”
- “We will need mobile.”
- “We will need teams.”
- “We will need integrations.”

Maybe you will.

But not all at once.

And maybe not in version one.

“Not yet” is not rejection.

“Not yet” is protection.

It lets you keep an idea without letting it invade the first build.

Use this sentence:

```text
For version one, we are not building __________, __________, or __________ yet.
```

Examples:

```text
For version one, we are not building payments, social sharing, or advanced 
analytics yet.
```

```text
For version one, we are not building team accounts, custom roles, or mobile 
notifications yet.
```

```text
For version one, we are not building recommendations, integrations, or an admin 
dashboard yet.
```

The “not yet” list is not a trash can.

It is a parking lot.

You are not throwing the ideas away. You are parking them where they cannot drive the first version off the road.

This list also helps when you later work with AI.

If you do not tell AI what is excluded, the tool may include normal-sounding features because they often appear in similar apps.

It may assume accounts are needed.

It may assume dashboards are needed.

It may assume settings are needed.

It may assume user roles are needed.

It may assume a larger app than you intended.

A “not yet” list helps you protect the first version from those assumptions.

Ask:

```text
Which future feature is most likely to sneak back into version one?
```

Write it down.

That is the feature to watch.

If everything feels too important to exclude, the idea is probably not ready to become a controlled project.

That does not mean the idea is bad.

It means the project boundary is not strong enough yet.

## 8. What Would Make This Project Too Expensive, Too Vague, or Too Large?

Some ideas should not become projects yet.

That does not mean they should die.

It means they need more clarity before they become real.

A stop sign is useful.

It tells you where the risk is before AI starts creating files, features, and assumptions around it.

The goal is not to scare you.

The goal is to help you build with your eyes open.

Use this sentence:

```text
This project is not ready if __________.
```

Here are common stop signs.

**Stop Sign 1: You cannot explain the problem simply**

If you need a long explanation before someone understands the problem, the project may still be too vague.

Try again:

```text
The problem is: __________.
```

If you cannot fill that in clearly, pause.

**Stop Sign 2: You cannot name the first user**

If version one is for “everyone,” it will probably become too large.

Try again:

```text
The first version is for __________.
```

If you cannot name that person or group clearly, pause.

**Stop Sign 3: Version one has too many jobs**

A first version should not solve five problems at once.

If your first version needs to manage, recommend, schedule, analyze, notify, share, sell, and automate, it is probably too large.

Try again:

```text
The first useful version only needs to let the user __________.
```

**Stop Sign 4: You cannot say what is not included yet**

If nothing can be excluded, the project has no boundary.

Try again:

```text
For version one, we are not building __________ yet.
```

If you cannot fill that in, pause.

**Stop Sign 5: The project depends on features you do not understand**

You do not need to be a professional developer.

But if version one depends on things you cannot explain, review, or recognize when broken, the project may be too risky for your current stage.

That does not mean you can never build it.

It means version one may need to be smaller.

**Stop Sign 6: You cannot review whether AI built the right thing**

If AI gives you output and you would not know whether it matches the intended project, pause.

AI can generate quickly.

You still need to judge the result.

If the project is too vague to review, it is too vague to build.

**Stop Sign 7: The idea feels exciting, but you cannot explain what “done” means**

A project needs an end point for version one.

Not forever.

Just version one.

If you cannot say what would count as a usable first version, AI may keep building past the point where you understand the project.

A warning light does not mean the car is worthless.

It means do not keep driving as if nothing is wrong.

The same is true here.

A stop sign does not mean the idea is bad.

It means the idea needs more clarity before it becomes real.

## 9. Ready, Pause, or Do Not Start Yet

You do not need certainty.

You need enough clarity to move forward with your eyes open.

At this stage, there are three useful decisions:

```text
Ready
Pause
Do Not Start Yet
```

All three can be good decisions.

### Ready

The idea may be ready to become a project if you can clearly answer:

```text
What is the idea?
What problem does it solve?
Who is version one for?
What does version one include?
What is not included yet?
What would make this project too expensive, vague, or large?
```

Ready does not mean guaranteed.

Ready does not mean the project will succeed.

Ready does not mean you have solved every future problem.

Ready means the idea is clear enough to become a controlled project.

It has a problem.

It has a first user.

It has a version-one boundary.

It has a “not yet” list.

It has enough shape that you can protect it.

### Pause

Pause may be the right decision if the idea is promising but one or more basic decisions are still unclear.

Maybe the problem is almost clear.

Maybe you know the user but not version one.

Maybe version one is close, but still too large.

Maybe the “not yet” list is missing.

Maybe you need to remove a few assumptions before the idea becomes real.

Pause is not failure.

Pause is how you avoid building confusion into the foundation.

### Do Not Start Yet

Do not start yet may be the right decision if the idea is too vague, too large, too expensive, or too dependent on assumptions to become a project right now.

That can be frustrating.

It can also save you from spending time and money on a project you cannot yet control.

Do not start yet does not mean “never.”

It means:

```text
This idea has not earned project status yet.
```

It can stay an idea.

You can keep notes.

You can revisit it.

You can simplify it.

You can define the problem more clearly.

You can choose a first user.

You can shrink version one.

You can build later.

The gate between idea and project is not there to block good ideas.

It is there to keep unclear projects from running ahead of you.

Before you move forward, make the call:

```text
My decision is: Ready / Pause / Do Not Start Yet

Because: __________.
```

If the answer is Ready, there is one more important thing to know.

The next step is still not “ask AI to build it.”

The next step is to design the project before the build starts.

That is enough to cross the first gate.
